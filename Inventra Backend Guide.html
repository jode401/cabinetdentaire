Inventra · Developer Handoff

Backend Developer Guide

How to turn the Inventra frontend prototype into a working product — what to edit, which APIs to build, and where the database fits.

Version 1.0 · Prepared for the backend engineer · Frontend file: Inventra.dc.html

1. What you've been given

The deliverable is a high-fidelity frontend prototype — a single self-contained HTML file (Inventra.dc.html) plus two small runtime helpers (support.js, image-slot.js). It renders the complete UI and all screen flows (marketing site, auth, onboarding, and the full dashboard app) and is fully interactive.

It is not yet wired to a backend. All data is mock/in-memory state held in the browser. Your job is to replace that mock state with real API calls and persistence. The visual layer, layout, states (loading, empty, error, success), and interactions are already designed — you should not need to redesign anything.

The one rule

Keep the screens, states, and flows as designed. Swap the data source underneath — don't rebuild the UI. Every empty state, loading skeleton and toast already exists for a reason.

4. Database — the core tables

Inventra is multi-tenant by store: a user (owner) can have one or more stores, and almost every record belongs to a store_id. Scope every query by the active store, and enforce it server-side.

-- core identity & tenancy users (id, full_name, email, password_hash, created_at) stores (id, owner_user_id → users, name, country, currency, tax_system, logo_url, accent_color, created_at) memberships (id, user_id → users, store_id → stores, role -- 'admin' | 'manager' | 'cashier', permission_level, created_at) -- catalog & inventory suppliers (id, store_id → stores, name, phone, email, total_purchases, last_purchase_at) products (id, store_id → stores, barcode, name, category, supplier_id → suppliers, cost_price, sell_price, quantity, reorder_threshold, created_at, updated_at) -- scanning & OCR pipeline documents (id, store_id → stores, uploaded_by → users, file_url, type -- 'invoice' | 'eod_report', status -- 'uploaded'|'processing'|'extracted'|'imported', created_at) ocr_lines (id, document_id → documents, detected_text, matched_product_id → products, detected_quantity, confidence, state -- 'pending'|'approved'|'rejected') -- movements, alerts, billing stock_moves (id, store_id, product_id → products, delta, reason -- 'scan_import'|'sale'|'manual'|'reorder', source_document_id → documents, created_at) sales (id, store_id, product_id → products, quantity, unit_price, sold_at) alerts (id, store_id, type, severity, product_id → products, message, status -- 'active'|'dismissed', created_at) subscriptions(id, store_id, plan, status, renews_at) invoices (id, store_id, amount, status, issued_at, pdf_url)
Don't store computed stock as the only source of truth

Prefer recording stock movements (stock_moves) and deriving products.quantity from them (or keep quantity as a cache you update in the same transaction). This gives you an audit trail — essential when an OCR import is later corrected or undone.

5. Where the database is actually needed

Not every screen needs its own tables. Here's where persistence is required vs. computed on the fly:

Needs tables (persist)

Products, suppliers, employees/memberships, documents, ocr_lines, stock_moves, sales, alerts, subscriptions, invoices, stores, users.

Computed / aggregated

Dashboard metrics, revenue/sales/expense/profit charts, low-stock lists, analytics cards. Derive these with SQL aggregates — don't store them.

Store-scoped (always filter)

Everything except users. Add store_id to every query and enforce membership server-side (row-level security if using Supabase).

Files (object storage, not DB)

Uploaded invoice PDFs/images and generated report/invoice PDFs. Store only the URL in the DB.

7. The OCR / scan pipeline (the signature feature)

This is the heart of Inventra. Build it as an asynchronous job so the UI can show the upload → processing → extracted states the prototype already renders.

  1. Upload — POST /documents stores the file in object storage, creates a documents row with status='uploaded', and enqueues an OCR job. Return the document id immediately.
  2. Extract — the worker sends the file to your OCR provider, parses line items (name, quantity, price), and writes one ocr_lines row per detected item with a confidence score. Set status='extracted'.
  3. Match — for each line, fuzzy-match detected_text against existing products (barcode first, then name similarity). Store the best match + confidence. Lines below ~80% confidence are flagged for review — the UI already shows this.
  4. Verify — the user approves/edits/rejects lines on the OCR screen (PATCH /ocr-lines/:id).
  5. Import — POST /documents/:id/import applies only approved lines: create/increment products and write stock_moves in one transaction, set status='imported'.
Confidence threshold

The UI treats < 80% as "needs review". Keep this configurable per store (there's already an Auto product detection toggle in Settings → AI that should gate auto-matching).

8. Practical tips & gotchas

  • Multi-store from day one. Even a single-store owner should go through the stores table. It saves a painful migration later.
  • Idempotent imports. A user may retry a failed import — key it by document_id so lines aren't applied twice.
  • Undo delete. The UI offers "Undo" on delete toasts. Implement soft-deletes (deleted_at) so undo is trivial.
  • Barcodes aren't unique globally. They're unique per store — put the unique constraint on (store_id, barcode).
  • Currency & tax are per store. Format money on the client using the store's currency; store raw numeric values (never formatted strings) in the DB.
  • Roles. Admin / Manager / Cashier already appear in the UI — enforce them server-side, not just by hiding buttons.
  • Autosave drafts. The Add-Product form autosaves a draft locally; you can keep that client-side or persist a draft per user.
  • Accent color & logo are per-store branding chosen at onboarding — persist on stores and return them with the session so the app themes itself.
